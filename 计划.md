# Shopify 独立站竞品分析平台需求分析报告

## 1. 项目目标

本项目用于本地分析 Shopify 独立站竞品商品。核心目标是：

- 录入 Shopify 独立站域名。
- 通过 `products.json` 初始化拉取商品基础信息。
- 使用 SQLite 持久化独立站、商品、价格、竞品关联、参数解析结果和报告。
- 支持设置唯一的我方独立站。
- 支持维护“我方商品 -> 多个竞品商品”的竞品关系。
- 在建立竞品关系时，针对竞品商品落地页调用 Gemini API 解析 ebike 参数并保存。
- 支持刷新价格、划线价、落地页参数。
- 支持导出我方商品的竞品分析报告，包含价格、参数和 LLM 生成的优劣势分析。

## 2. 关键约束与可行性判断

### 2.1 products.json 的定位

`products.json` 适合获取稳定的商品基础数据：

- `handle`
- `title`
- `price`
- `compare_at_price`
- `product_type`
- `vendor`

本项目第一版只强依赖：

- `handle`
- `price`
- `compare_at_price`
- `vendor`

落地页 URL 通过以下规则生成：

```text
https://{domain}/products/{handle}
```

其中 `handle` 的主要作用是拼接商品落地页 URL，便于后续抓取 HTML 并解析 `Specification` 等参数区块。

### 2.2 参数数据的定位

ebike 参数通常不在 `products.json` 里，而是在商品落地页 HTML 中，例如：

- `Specification`
- `Specifications`
- `Specs`
- `Technical Specs`
- `Product Details`
- 参数卡片
- 商品描述富文本
- 页面内嵌 JSON

所以参数解析流程应以“抓取商品落地页 HTML -> 提取规格区块 -> Gemini 标准化”为主。

### 2.3 关于“不需要 Java 后端”的技术说明

如果是纯浏览器 React 应用，会遇到三个现实限制：

1. 浏览器直接请求竞品站 HTML / `products.json` 可能遇到 CORS 限制。
2. 浏览器不能直接写入普通本地 SQLite 文件。
3. Gemini API Key 放在前端代码里会暴露。

因此这里的“不需要后端”应理解为：

```text
不需要 Java / Spring Boot 这类重后端
不需要部署线上后端服务
```

而不是：

```text
完全没有本地能力层
```

推荐采用本地应用方案：

```text
React + Vite 前端
+ 本地能力层
+ SQLite
+ Gemini API
```

本地能力层可以是：

- 一个随 `npm run dev` 一起启动的本地 Node.js 进程
- Python FastAPI / Flask 本地进程
- Electron main process
- Tauri command

第一版建议使用：

```text
React + Vite + 本地 Node.js 能力层 + SQLite
```

这样仍然是本地工具，不需要部署后端服务，同时可以解决跨域抓取、SQLite 写入和密钥管理问题。

本地能力层只承担四类事情：

- 读取和写入 SQLite。
- 请求 Shopify `products.json` 和商品落地页 HTML。
- 读取本地环境变量中的 Gemini API Key 并调用 Gemini。
- 将结构化结果返回给 React 页面。

它不是业务意义上的重后端，而是本地工具的系统能力层。

## 3. 用户角色与核心场景

### 3.1 用户角色

当前阶段只有一个用户：

- 本地使用者
- 负责录入竞品站、维护竞品关系、确认参数和导出报告

### 3.2 核心场景

1. 录入 Shopify 独立站域名。
2. 系统拉取该域名下所有商品。
3. 设置其中一个域名为我方独立站。
4. 选择我方商品。
5. 选择竞品站。
6. 在竞品站商品中模糊搜索并选择竞品商品。
7. 建立竞品关系。
8. 系统抓取竞品商品落地页 HTML。
9. 调用 Gemini 解析该竞品商品参数。
10. 将参数保存到 SQLite。
11. 后续选择我方商品导出竞品分析报告。
12. 报告中展示我方商品与所有关联竞品商品的价格、参数和优劣分析。

## 4. 功能需求

### 4.1 独立站管理

功能：

- 新增独立站域名。
- 新增独立站时允许输入商品最低价格过滤值。
- 若未输入最低价格，默认使用 `300`。
- 域名标准化，例如：

```text
qlifebike.com
https://qlifebike.com
https://www.qlifebike.com/
```

统一保存为：

```text
https://www.qlifebike.com
```

- 查看已录入站点列表。
- 手动刷新某个站点商品。
- 设置某个站点为我方独立站。
- 同一时间只能有一个我方独立站。
- 设置新我方站点前，必须先取消当前我方站点。

### 4.2 商品初始化

录入域名后，系统调用：

```text
https://{domain}/products.json?limit=250&page=1
```

分页拉取商品，直到返回为空或不足一页。

同步时需要执行商品过滤：

- 只保留 `price > 最低价格过滤值` 的商品。
- 最低价格过滤值由录入站点时填写，未填写时默认为 `300`。
- 排除 `vendor == "Seel"` 的保险商品。
- 被过滤掉的商品不进入第一版商品列表，也不参与竞品关系维护。

保存字段：

- 站点 ID
- Shopify 商品 ID
- handle
- title
- price
- compare_at_price
- vendor
- landing_page_url
- raw_json
- last_price_synced_at

第一版重点保存：

- `handle`
- `price`
- `compare_at_price`
- `vendor`
- `landing_page_url`

### 4.3 我方主体设置

规则：

- 只能有一个站点 `is_own_site = true`。
- 若已经存在我方站点，设置其他站点前需要先取消原有设置。
- 页面上明确显示当前我方站点。

### 4.4 竞品关系维护

操作流程：

1. 选择我方站点的一款商品。
2. 点击“添加竞品”。
3. 选择竞品独立站。
4. 展示该竞品站所有商品。
5. 支持按标题、handle 模糊查询。
6. 选择一个竞品商品。
7. 建立关联关系。
8. 建立关系后触发竞品商品参数解析。

关系规则：

- 一个我方商品可以关联多个竞品商品。
- 一个竞品商品可以被多个我方商品复用。
- 若竞品商品已有较新的参数解析结果，可提示是否复用或刷新。

### 4.5 参数解析

触发场景：

- 建立竞品关系时自动解析竞品参数。
- 商品详情页手动点击“刷新参数”。
- 导出报告前选择“刷新所有竞品参数”。

解析输入：

- 商品标题
- 当前价格
- 划线价
- handle
- 落地页 URL
- 商品落地页 HTML
- 从 HTML 初步提取的规格区块文本

解析流程：

1. 抓取落地页 HTML。
2. 优先提取以下区块：

```text
Specification
Specifications
Specs
Technical Specs
Product Details
Details
Geometry
Components
```

3. 提取表格、列表、参数卡片和 key-value 文本。
4. 将提取文本发送给 Gemini。
5. Gemini 只负责标准化，不允许无证据猜测。
6. 返回固定 JSON schema。
7. 写入 SQLite。
8. 标记参数来源、置信度和解析时间。

### 4.6 ebike 参数固定排序

所有商品参数按固定顺序展示。缺失参数显示为空或 `未获取`。

建议顺序：

1. 商品名称
2. 商品类型

18. 显示屏
19. 充电器
20. 充电时间
21. 车架材质
22. 前叉
23. 后避震
24. 刹车类型
25. 轮胎尺寸

34. 防水等级
35. 认证标准
36. 质保
37. 其他卖点

### 4.7 参数冲突处理

同一页面可能出现冲突，例如：

- 参数卡片写 `80Nm`
- Specification 写 `100Nm`

保存策略：

```json
{
  "selected_value": "100Nm",
  "sources": [
    {
      "value": "100Nm",
      "source": "Specification",
      "confidence": 0.9
    },
    {
      "value": "80Nm",
      "source": "Highlight Specs",
      "confidence": 0.75
    }
  ],
  "conflict": true
}
```

展示策略：

- 默认展示 `selected_value`。
- 若存在冲突，显示冲突标记。
- 支持人工确认最终值。

### 4.8 数据刷新

价格刷新：

- 重新调用 `products.json`。
- 更新 `price` 和 `compare_at_price`。
- 不覆盖人工维护的竞品关系。

参数刷新：

- 重新抓取落地页 HTML。
- 重新调用 Gemini 解析。
- 生成新的解析版本。
- 默认使用最新版本。
- 保留历史解析记录，便于追溯。

报告前刷新：

- 导出竞品分析报告前提供选项：

```text
仅使用已保存参数
刷新我方商品参数
刷新所有竞品参数
刷新价格和参数
```

第一版建议默认：

```text
刷新价格，参数由用户手动决定是否刷新
```

### 4.9 竞品分析报告

输入：

- 一个我方商品。

选择一个我方商品后，系统自动查询该商品已维护的所有竞品商品。

报告生成前的竞品选择：

- 默认列出该我方商品关联的全部竞品商品。
- 支持单独勾选参与本次报告的竞品商品。
- 支持“全选”按钮。
- 支持“取消全选”。
- 未勾选的竞品不参与本次报告生成。

报告生成时使用：

- 我方商品基础信息、价格、划线价和参数。
- 被选中的竞品商品基础信息、价格、划线价和参数。
- 参数最后解析时间。
- 参数冲突和缺失状态。

输出：

- 一个横向参数对比表。
- 表格中包含我方商品参数列。
- 表格中包含每个被选中竞品商品的参数列。
- 表格中包含每个竞品相对我方商品的 LLM 优劣势分析列。
- 缺失参数提示。
- 参数冲突提示。
- LLM 生成的逐竞品优劣势分析。
- LLM 生成的整体竞争态势总结。
- 建议销售话术或差异化卖点。

报告内容结构：

```text
1. 报告摘要
2. 我方商品信息
3. 可选竞品商品列表
4. 本次参与分析的竞品商品
5. 价格与核心参数对比表
6. 我方商品 vs 每个竞品商品的优劣势分析
7. 整体竞争态势总结
8. 可优化建议
9. 数据可信度说明
```

参数对比表建议结构：

```text
参数项 | 我方商品 | 竞品 A | 竞品 A 分析 | 竞品 B | 竞品 B 分析
价格 | $999 | $1199 | 我方价格更低，但若竞品电池更大需结合续航说明 | $899 | 我方价格偏高，需要强调配置优势
电机额定功率 | 750W | 750W | 功率持平 | 500W | 我方动力优势明显
电池容量 | 48V 15Ah | 48V 20Ah | 我方电池容量弱于竞品 | 48V 12Ah | 我方电池容量更强
```

其中“竞品 X 分析”由 LLM 根据当前参数行或核心差异生成，重点说明我方相对该竞品的优势、劣势和可用于营销的话术方向。

## 5. 数据库设计

### 5.1 sites

独立站表。

```sql
CREATE TABLE sites (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  domain TEXT NOT NULL UNIQUE,
  origin TEXT NOT NULL,
  name TEXT,
  is_own_site INTEGER NOT NULL DEFAULT 0,
  min_product_price INTEGER NOT NULL DEFAULT 30000,
  last_product_sync_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

`min_product_price` 使用最小货币单位保存，例如 `30000` 表示 `$300.00`。

### 5.2 products

商品表。

```sql
CREATE TABLE products (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  site_id INTEGER NOT NULL,
  shopify_product_id TEXT,
  handle TEXT NOT NULL,
  title TEXT,
  vendor TEXT,
  price INTEGER,
  compare_at_price INTEGER,
  currency TEXT DEFAULT 'USD',
  landing_page_url TEXT NOT NULL,
  raw_json TEXT,
  last_price_sync_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE(site_id, handle),
  FOREIGN KEY(site_id) REFERENCES sites(id)
);
```

价格建议以最小货币单位保存，例如美分。

### 5.3 competitor_relations

竞品关系表。

```sql
CREATE TABLE competitor_relations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  own_product_id INTEGER NOT NULL,
  competitor_product_id INTEGER NOT NULL,
  note TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE(own_product_id, competitor_product_id),
  FOREIGN KEY(own_product_id) REFERENCES products(id),
  FOREIGN KEY(competitor_product_id) REFERENCES products(id)
);
```

### 5.4 product_pages，可选

落地页 HTML 缓存表。

注意：`product_pages` 不是为了保存落地页 URL。落地页 URL 已经通过 `products.landing_page_url` 保存。

这个表的意义是保存“某次抓取到的 HTML 快照”，用于：

- 避免短时间内重复请求竞品站。
- 记录参数解析时使用的是哪一次页面内容。
- 调试 Gemini 参数抽取结果。
- 对比参数刷新前后的页面变化。

第一版如果想保持数据库简单，可以不建 `product_pages` 表。此时流程改为：

- `products` 表保存 `landing_page_url`。
- 参数解析时实时抓取 HTML。
- 将提取后的规格文本保存到 `spec_extractions.input_text`。
- 将 Gemini 原始输出保存到 `spec_extractions.output_json`。

如果后续需要追溯原始 HTML，再增加该表。

```sql
CREATE TABLE product_pages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  product_id INTEGER NOT NULL,
  url TEXT NOT NULL,
  html TEXT,
  fetched_at TEXT NOT NULL,
  FOREIGN KEY(product_id) REFERENCES products(id)
);
```

### 5.5 spec_extractions

参数解析任务表。

```sql
CREATE TABLE spec_extractions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  product_id INTEGER NOT NULL,
  provider TEXT NOT NULL,
  model TEXT,
  input_text TEXT,
  output_json TEXT,
  status TEXT NOT NULL,
  error_message TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY(product_id) REFERENCES products(id)
);
```

### 5.6 product_specs

标准参数表。

```sql
CREATE TABLE product_specs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  product_id INTEGER NOT NULL,
  spec_key TEXT NOT NULL,
  spec_label TEXT NOT NULL,
  spec_order INTEGER NOT NULL,
  value TEXT,
  unit TEXT,
  raw_text TEXT,
  source_type TEXT,
  confidence REAL,
  conflict INTEGER NOT NULL DEFAULT 0,
  manually_verified INTEGER NOT NULL DEFAULT 0,
  extraction_id INTEGER,
  updated_at TEXT NOT NULL,
  UNIQUE(product_id, spec_key),
  FOREIGN KEY(product_id) REFERENCES products(id),
  FOREIGN KEY(extraction_id) REFERENCES spec_extractions(id)
);
```

### 5.7 reports

报告表。

```sql
CREATE TABLE reports (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  own_product_id INTEGER NOT NULL,
  content_markdown TEXT,
  input_snapshot_json TEXT,
  provider TEXT,
  model TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY(own_product_id) REFERENCES products(id)
);
```

## 6. Gemini 调用设计

### 6.1 本地能力层与 SQLite 实现方案

本项目不需要 Java 后端，但需要一个本地能力层来处理浏览器不适合直接做的事情。

可选方案如下。

#### 方案 A：Node.js 本地 API，推荐第一版使用

结构：

```text
React + Vite
  -> http://localhost:{port}/api/*
    -> Node.js
      -> SQLite
      -> Shopify 页面抓取
      -> Gemini API
```

优点：

- 和 React/Vite 同属 JavaScript/TypeScript 技术栈。
- 可以通过 `npm run dev` 同时启动前端和本地 API。
- SQLite、HTTP 请求、Gemini SDK 都容易接入。
- 后续如果要迁移 Electron，也比较自然。

适合第一版。

推荐依赖：

```text
better-sqlite3 或 sqlite
express 或 fastify
zod
@google/generative-ai 或 Google GenAI SDK
```

推荐脚本：

```json
{
  "scripts": {
    "dev": "concurrently \"npm run dev:web\" \"npm run dev:local\"",
    "dev:web": "vite",
    "dev:local": "tsx server/index.ts"
  }
}
```

#### 方案 B：Python 本地 API

结构：

```text
React + Vite
  -> http://localhost:{port}/api/*
    -> Python FastAPI / Flask
      -> SQLite
      -> Shopify 页面抓取
      -> Gemini API
```

优点：

- Python 处理 HTML、文本抽取、数据清洗比较方便。
- `sqlite3` 是标准库，落地快。
- 如果后续需要引入更多爬虫、解析、数据分析能力，Python 有优势。

缺点：

- 前端和本地能力层变成两套语言环境。
- `npm run dev` 需要额外管理 Python 虚拟环境。

适合后续做更复杂的数据解析时考虑。

#### 方案 C：Electron

结构：

```text
Electron Renderer
  -> Electron Main
    -> SQLite
    -> Shopify 页面抓取
    -> Gemini API
```

优点：

- 更像真正的桌面应用。
- 可以直接访问本地文件系统。
- API Key 和 SQLite 都可以放在 main process。

缺点：

- 项目复杂度比普通 Vite 高。
- 第一版学习成本略高。

适合在 MVP 跑通后再升级。

#### 方案 D：Tauri

优点：

- 桌面应用体积比 Electron 小。
- 本地能力强。

缺点：

- 需要 Rust 生态参与。
- 对当前 React/Vite 学习目标不如 Node.js 顺滑。

第一版不建议优先选择。

第一版最终建议：

```text
React + Vite + Node.js 本地 API + SQLite + Gemini API
```

这不是 Java 后端，也不需要线上部署；它只是本地工具为了抓网页、写数据库、保护密钥所需的轻量能力层。

### 6.2 本地 API 职责划分

React 前端负责：

- 页面展示。
- 表单输入。
- 站点、商品、竞品关系的交互。
- 参数和报告结果展示。

Node.js 本地 API 负责：

- 初始化 SQLite。
- 增删改查站点。
- 调用 Shopify `products.json` 同步商品。
- 根据 `handle` 拼接落地页 URL。
- 抓取商品落地页 HTML。
- 提取疑似规格区块。
- 调用 Gemini API。
- 保存参数解析结果。
- 生成和保存竞品分析报告。

### 6.3 推荐本地 API 路由

```text
GET    /api/health

GET    /api/sites
POST   /api/sites
PATCH  /api/sites/:id
POST   /api/sites/:id/sync-products
POST   /api/sites/:id/set-own
POST   /api/sites/:id/unset-own

GET    /api/products
GET    /api/products/:id
POST   /api/products/:id/refresh-price
POST   /api/products/:id/fetch-page
POST   /api/products/:id/extract-specs

GET    /api/relations
POST   /api/relations
DELETE /api/relations/:id

POST   /api/reports
GET    /api/reports/:id
```

创建报告时请求体建议：

```json
{
  "ownProductId": 1,
  "selectedCompetitorProductIds": [11, 12, 13],
  "refreshPrice": true,
  "refreshSpecs": false
}
```

其中 `selectedCompetitorProductIds` 来自报告页勾选结果；全选时传入该我方商品关联的全部竞品商品 ID。

### 6.4 Shopify 数据同步调用方式

Node.js 本地 API 负责请求：

```text
https://{domain}/products.json?limit=250&page=1
```

同步流程：

1. 按页请求 `products.json`。
2. 读取 `handle`、`title`、`price`、`compare_at_price`、`vendor`。
3. 过滤 `price > min_product_price`。
4. 过滤 `vendor == "Seel"`。
5. 拼接 `landing_page_url`。
6. upsert 到 SQLite。

### 6.5 商品落地页 HTML 抓取方式

Node.js 本地 API 负责请求：

```text
https://{domain}/products/{handle}
```

抓取后从 HTML 中初步提取：

- 页面标题。
- 商品描述。
- `Specification` 附近文本。
- table。
- list。
- 参数卡片。
- `application/ld+json`。
- 页面内嵌 product JSON。

第一版可以不保存完整 HTML，只保存本次传给 Gemini 的规格相关文本到 `spec_extractions.input_text`。如果需要页面快照追溯，再启用 `product_pages` 缓存表保存完整 HTML。

前端不直接抓竞品页面，避免 CORS 问题。

### 6.6 Gemini API Key 保存

第一版本地工具可以将 Gemini API Key 保存到本地配置中，例如：

```text
.env.local
```

不建议提交到 Git。

示例：

```text
GEMINI_API_KEY=your_api_key
GEMINI_MODEL=gemini-2.5-flash
```

Node.js 本地 API 从环境变量读取，不返回给前端页面。

### 6.7 Gemini 调用方式

Gemini 调用由 Node.js 本地 API 统一封装，前端只调用本地接口：

```text
POST /api/products/:id/extract-specs
```

后端内部流程：

1. 从 SQLite 读取商品。
2. 若没有最新 HTML，则抓取落地页 HTML。
3. 从 HTML 提取规格相关文本。
4. 构造 Gemini prompt。
5. 调用 Gemini。
6. 校验 Gemini 返回 JSON。
7. 写入 `spec_extractions` 和 `product_specs`。
8. 返回标准化参数给前端。

### 6.8 参数抽取 Prompt 原则

要求：

- 只从输入文本中抽取。
- 不允许猜测。
- 没有证据返回 `null`。
- 固定 schema。
- 所有参数按固定顺序输出。
- 保留 `raw_text`。
- 标记置信度。
- 标记冲突。

### 6.9 参数抽取输出示例

```json
{
  "specs": [
    {
      "key": "motor_peak_power",
      "label": "电机峰值功率",
      "order": 7,
      "value": "3500W",
      "unit": "W",
      "raw_text": "Peak 3500W mid-drive motor",
      "source_type": "specification",
      "confidence": 0.9,
      "conflict": false
    }
  ]
}
```

## 10. 风险与应对

### 10.1 CORS 风险

风险：

纯前端无法稳定请求竞品站 HTML。

应对：

使用本地能力层发起请求，不走浏览器跨域限制。

### 10.2 参数不统一

风险：

不同站点规格写法不同，同一页面也可能参数冲突。

应对：

固定 schema + source + confidence + conflict + 人工确认。

### 10.3 LLM 输出不稳定

风险：

Gemini 输出可能不是合法 JSON，或者出现推测。

应对：

强 prompt 约束 + JSON schema 校验 + 失败重试 + 人工确认。

### 10.4 商品价格变化

风险：

竞品价格频繁变化。

应对：

报告生成前默认刷新价格。

### 10.5 参数过期

风险：

参数在建立竞品关系后发生变化。

应对：

报告生成前提供参数刷新选项，并显示最后解析时间。

## 12. 结论

本项目可行，但不建议做成纯浏览器前端。原因是抓 Shopify 落地页 HTML、写 SQLite、保护 Gemini API Key 都需要本地能力层。这里的能力层推荐使用 Node.js 本地 API，而不是 Java/Spring 这类重后端。

最合理的产品形态是：

```text
本地 React/Vite 竞品分析工具
+ 本地 SQLite 数据库
+ Node.js 本地 API
+ 本地抓取 Shopify 页面
+ Gemini 辅助参数标准化
+ 人工确认
+ Markdown 报告导出
```

第一版应优先跑通完整闭环：

```text
录入站点 -> 同步商品 -> 设置我方站点 -> 建立竞品关系 -> 解析竞品参数 -> 生成竞品报告
```
